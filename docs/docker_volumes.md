# Docker Volumes: Anonymous, Named, and Bind Mounts

You mentioned having issues with your server disk filling up from anonymous volumes generated by cron jobs. This is a very common Docker trap! Let's break down the three types of file mapping in Docker, what they do, and why I used what I did in `compose.yaml`.

## 1. Anonymous Volumes
**Looks like:** `- /app/data` (only one path provided) or defined via a `VOLUME /app/data` instruction inside a `Dockerfile`.

*   **How it works:** Docker creates a hidden folder inside `/var/lib/docker/volumes/` with a long randomized hash (e.g., `5e8b6..._data`).
*   **The Disk Space Trap:** If you run `docker run my-image`, Docker creates an anonymous volume. If you stop and remove that container (`docker rm my-container`), **Docker does NOT delete the anonymous volume by default**. It becomes a "dangling volume". If a cron job runs repeatedly and finishes, and leaves behind anonymous volumes, your disk will fill up incredibly fast.
    *   *Note: `docker run --rm` is specifically designed to clean up associated anonymous volumes upon exit, but if the cron job ever fails to append `--rm`, or if the service crashes, they linger.*
    *   *Fix:* Running `docker volume prune` periodically cleans these up.

## 2. Named Volumes
**Looks like:** `- my_db_data:/var/lib/postgresql/data` (A name on the left side, NO slashes).

*   **How it works:** Docker creates a folder in `/var/lib/docker/volumes/my_db_data`. 
*   **Why use it?** Unlike anonymous volumes, these are explicit. If you destroy your database container and start a new one pointing to `my_db_data`, all your data is instantly restored. Docker handles the file permissions seamlessly. This is the **gold standard** for Production Databases (PostgreSQL, Redis, etc.).

## 3. Bind Mounts (What we used)
**Looks like:** `- ./models:/app/models` (A real file path or `./` on the left side).

*   **How it works:** It literally punches a hole through the container into your host machine's exact directory. If the container writes a file to `/app/models`, you immediately see it in your project's `./models` folder on your host OS.
*   **Why use it here?** Machine Learning models are huge (800MB+), and we want to actually *see* and *cache* them natively in our repository folder structure (hence adding it to `.gitignore`). If we used a Named Volume, the model would be hidden deep in Docker's internal linux system files. A bind mount ensures you can easily wipe `./models` from your terminal if you want to clear the cache, or manually drop files into it without entering the container.

---

### Summary Checklist:
1. **Bind Mounts (`./data:/app/data`)**: Best for config files, live-reloading code (dev mode), or caching ML models you want explicit host access to.
2. **Named Volumes (`postgres_data:/var/lib/db`)**: Best for databases and persistent app state that you don't need to manually browse on the host. 
3. **Anonymous Volumes**: Generally **avoid** unless you specifically need temporary, disposable high-speed I/O that doesn't bloat the container's writable layer, and you rigidly manage cleanup.
